Executive Summary

The Family Dashboard is a self-hosted web application that acts as a single control center for your household. Each family member gets a personalized, rearrangeable dashboard populated with both built-in apps (grocery lists, to-dos, chore tracker) and live embedded cards pointing to any external site or service.

Because this lives entirely on your local network — protected by Tailscale — there is no reliance on any cloud service for the core experience. The application can run on anything from a Raspberry Pi to a home server or NAS, and family members can access it securely from any device, anywhere, via the Tailscale VPN.

Design Philosophy: Keep it simple to self-host, fast to use daily, and easy to extend. The architecture is intentionally lean — one database file, one Node.js process — packaged as a Helm chart deployed onto a local k3s cluster, with full responsive support from mobile to TV.

Recommended Tech Stack
Frontend Framework ✓ Recommended
React 19 + TypeScript
Component-based architecture is a natural fit for a card/widget dashboard. TypeScript eliminates whole classes of runtime bugs and is essential for a long-lived family project. Vite provides near-instant hot reload during development.
React 19 TypeScript Vite
Dashboard Grid ✓ Recommended
React Grid Layout
The industry-standard library for draggable, resizable, responsive grid layouts in React. Supports saving/restoring layouts per user, breakpoints for mobile vs. desktop, and full-screen card expansion. Zero additional dependencies.
react-grid-layout drag & drop resize
Styling ✓ Recommended
Tailwind CSS v4
Utility-first CSS is ideal for building consistent card UIs quickly. Tailwind v4's improved performance and CSS variable system makes theming per-profile straightforward. No CSS file bloat over time.
Tailwind CSS v4 shadcn/ui Framer Motion
State Management ✓ Recommended
Zustand + TanStack Query
Zustand handles client-side state (active profile, layout state, UI state) with minimal boilerplate. TanStack Query manages all server data fetching, caching, and background refresh — critical for keeping calendar and weather cards current.
Zustand TanStack Query v5
Backend ✓ Recommended
Node.js + Fastify
Fastify is 2–3× faster than Express with built-in schema validation and TypeScript support out of the box. It handles REST API routes, WebSocket connections for real-time list sync, and acts as a proxy for Google/weather APIs to keep secrets server-side.
Node.js 22 LTS Fastify WebSockets
Database ✓ Recommended
SQLite + Drizzle ORM
SQLite is the perfect choice for local hosting — zero server setup, single file, trivially backed up. Drizzle ORM provides type-safe queries with automatic TypeScript inference, schema migrations, and is purpose-built for SQLite. No Postgres server to maintain.
SQLite (better-sqlite3) Drizzle ORM WAL mode
Deployment & Orchestration ✓ Recommended
k3s + Helm
k3s is a lightweight, production-grade Kubernetes distribution that runs comfortably on a home server or NAS. The app is packaged as a Helm chart with a values.yaml for easy configuration. Traefik (bundled with k3s) handles ingress and TLS. A single helm upgrade --install deploys or updates the app.

Tailscale creates an encrypted WireGuard mesh network between all your family's devices. No ports to open on your router, no dynamic DNS to manage. Each device gets a stable Tailscale IP. The dashboard is only reachable from trusted, authenticated devices.

Feature Design
👤
Profile System
Each family member has a named profile with an avatar, color theme, and their own dashboard layout. Profiles are either Public (switchable freely) or Private (require a numeric PIN to switch to). Private profiles hide their private-marked cards from other profiles. The active profile is stored in a short-lived session cookie so it persists through page refreshes.
PIN stored as bcrypt hash — never plaintext
⊞
Dashboard Grid Engine
Built on React Grid Layout, each card can be dragged to any position and resized by its handle. A "maximize" button expands any card to fill the viewport. Clicking outside or pressing Escape returns it to its original position. Layout is saved per profile to the database on every change — no manual save button needed.
Separate desktop and mobile layout breakpoints
🔗
External Link Cards (iFrame Embeds)
Any family member can add a card that embeds an external URL via an <iframe>. When a site blocks iFraming (X-Frame-Options), the card gracefully falls back to a stylized link preview with favicon, title, and description fetched via the server-side proxy. Cards can be marked as shared (visible to all profiles) or private to just the creator.
Server proxy fetches oEmbed/Open Graph for preview data
📅
Google Calendar Integration
The calendar card uses the Google Calendar API with OAuth 2.0. The server stores the refresh token securely and fetches events on behalf of the user — keeping API keys entirely off the client. Events from multiple calendars can be merged and color-coded. The card shows a week view inline and expands to a full month view when maximized.
OAuth tokens stored encrypted in the database
🌤
Weather Card
Fetches current conditions and a 7-day forecast from the OpenWeatherMap API (free tier). Location can be set per profile. The server caches weather data for 30 minutes to avoid hitting API rate limits. The card shows current conditions in compact view and hourly forecast when expanded.
30-min server-side cache — stays within free tier limits
✅
Built-in To-Do List
A full-featured task manager with lists, due dates, priorities, and assignees. Tasks can be assigned to specific profiles, making it easy to delegate chores or family responsibilities. Completed tasks are archived rather than deleted. Shared lists are visible across all profiles; personal lists are private by default. Real-time sync via WebSocket means changes show up instantly on other family members' screens.
WebSocket push — no need to refresh for updates
🛒
Grocery List
A shared grocery list that any family member can add to. Items are categorized (produce, dairy, pantry, etc.) and can be checked off as you shop. Checked items are sorted to the bottom and cleared with a single "Done Shopping" button. The list supports a simple history so frequently bought items can be re-added quickly. Fully real-time via the same WebSocket channel as to-dos.
🧹
Chore Tracker (External Embed)
Since the chore tracker is an existing separate project, it is embedded as an iFrame card. The card passes the active profile name as a URL parameter so the chore tracker can pre-filter to the relevant family member's chores. If the chore tracker is also on the local network, this works seamlessly within the Tailscale tunnel.
Profile context passed via URL query params

Security Model
🔒 Tailscale VPN Layer
The app is never exposed to the public internet. Tailscale creates a private WireGuard mesh — only devices enrolled in your Tailnet can even reach the server. This replaces the need for public-facing auth entirely and makes the app safe to run without HTTPS (though we still recommend it).
🔑 PIN-based Profile Switching
Private profiles require a 4–6 digit PIN to switch to. PINs are stored as bcrypt hashes (cost factor 12). A failed PIN increments a rate-limit counter — after 5 failures, switching is blocked for 5 minutes. This prevents brute-force guessing by a family member.
🗝️ API Secrets Stay Server-Side
Google OAuth tokens, OpenWeatherMap API keys, and any other third-party credentials are stored only in the server's environment and encrypted in the database. The browser never receives these values — all external API calls are proxied through the Fastify backend.
🛡️ Content Privacy
Cards and list data marked as private are filtered server-side before being sent to the client — they never appear in any API response unless the requesting session matches the owning profile. This isn't just UI-level hiding; private content is excluded at the query layer.

Responsive Design Strategy

Supporting desktop, tablet, and mobile is not just a CSS concern — it changes how the dashboard grid behaves, how cards are controlled, and what the navigation pattern looks like at each size. React Grid Layout's built-in breakpoint system handles most of the heavy lifting, but there are several important design decisions per device class.

🖥
Desktop (lg / xl — ≥996px)
Full drag-and-drop grid. Cards can be freely positioned anywhere on a 12-column grid, resized by dragging a corner handle, and maximized to fill the viewport. Profile switcher lives in a persistent left sidebar. Card controls are always visible in the card header. This is the primary layout experience — everything else is an adaptive simplification of it.
12-column grid · drag + resize enabled
⬛
Tablet (md / sm — 480px–995px)
Switches to a 6-column grid. Drag-and-drop remains enabled. The sidebar collapses to a top navigation bar with a profile avatar button. Card controls condense — resize handles remain, but the settings and hide buttons move into a tap-friendly overflow menu (three-dot) per card. Landscape tablet gets close to the desktop experience; portrait tablet gets the condensed layout.
6-column grid · drag retained · condensed controls
📱
Mobile (xs — <480px)
Cards stack in a single full-width column. Drag-and-drop is replaced by a dedicated "Edit Layout" mode where cards can be long-pressed and reordered via up/down handles — this is far more reliable on touch than free-form dragging. Card controls are accessible via a bottom sheet that slides up on tap. The profile switcher becomes a bottom navigation bar with avatar chips. All tap targets are minimum 44×44px per Apple/Google guidelines.
1-column stacked · reorder mode instead of drag · bottom nav
📲
PWA — Add to Home Screen
The app ships with a full PWA manifest and service worker, so family members can install it to their phone or tablet home screen and launch it in standalone mode (no browser chrome). On iOS this uses the "Add to Home Screen" flow; on Android it triggers the native install prompt. Offline support is limited to showing the last cached layout — live data requires connectivity.
Standalone mode · app icon · splash screen per profile color
🖼
iFrame Cards on Mobile
iFrame embeds are particularly tricky on mobile — many sites are not mobile-optimized when embedded, and touch scroll is intercepted by the iframe. The card shell detects touch devices and adds pointer-events: none to iFrames in compact mode, with a tap-to-activate overlay that hands control to the iframe once the user deliberately taps in. Maximized iFrames get full touch passthrough.
Tap-to-activate prevents accidental iframe scroll capture
Layout Storage: Each profile stores a separate layout object per breakpoint — mobile, tablet, and desktop layouts are independent. Rearranging cards on your phone doesn't disturb your desktop layout.

Kubernetes / Helm Deployment

The application is packaged as a Helm chart targeting a k3s cluster. k3s is a certified Kubernetes distribution purpose-built for resource-constrained environments — it runs comfortably on a Raspberry Pi 4, an old Mac Mini, a NAS, or any always-on home server. A full install takes under a minute.

Important SQLite constraint: Because SQLite uses a single file with exclusive write locks, the app Deployment must run with replicas: 1. The static frontend pod (nginx serving the React build) can scale freely. If horizontal scaling of the backend ever becomes necessary, a migration to PostgreSQL is straightforward with Drizzle's migration system — the schema stays the same.
Helm Chart Structure
family-dashboard/ # Helm chart root
├── Chart.yaml
├── values.yaml # storageClass, domain, image tags, replicas
└── templates/
    ├── deployment-app.yaml # Node.js/Fastify · replicas: 1
    ├── deployment-static.yaml # nginx static · replicas: 2
    ├── service-app.yaml
    ├── service-static.yaml
    ├── ingress.yaml # Traefik IngressRoute + TLS
    ├── pvc.yaml # ReadWriteOnce · SQLite data volume
    ├── configmap.yaml # App config, feature flags
    ├── secret.yaml # Google OAuth, Weather API key
    └── cronjob-backup.yaml # Nightly SQLite → host path backup
Key values.yaml Parameters
app:
  image: family-dashboard/app:latest
  replicas: 1 # SQLite — must stay 1
  port: 3001
static:
  image: family-dashboard/static:latest
  replicas: 2
ingress:
  host: dashboard.home.ts.net # Tailscale MagicDNS hostname
  tls: true
persistence:
  storageClass: local-path # k3s built-in local-path provisioner
  size: 2Gi
backup:
  schedule: "0 2 * * *" # 2am nightly
  hostPath: /mnt/backups/family-dashboard
Tailscale Kubernetes Operator

Rather than exposing a NodePort or configuring a LoadBalancer, we use the Tailscale Kubernetes Operator. This runs as a pod inside the cluster and registers the service directly onto your Tailnet — no port forwarding, no firewall rules. Family members access the dashboard at a stable MagicDNS address like dashboard.home.ts.net, which resolves correctly whether they're on the home network or remote via Tailscale.

Deploy command: helm upgrade --install family-dashboard ./family-dashboard -f values.yaml -n family --create-namespace — a single command installs or updates the entire application.
